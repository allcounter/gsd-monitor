# Roadmap: GSD Monitor

## Overview

GSD Monitor transforms from an empty SwiftUI canvas into a live macOS dashboard through five foundational phases. We begin by establishing Swift 6 concurrency patterns and data models, add file discovery and parsing capabilities, integrate FSEvents for real-time monitoring, build the visual dashboard interface, and finally polish with notifications and editor integration. Each phase validates critical architecture decisions before adding complexity, following a bottom-up approach that addresses memory leaks, concurrency safety, and filesystem monitoring pitfalls early.

## Phase Details

### Phase 1: Foundation & Models
**Goal**: Establish Swift 6 concurrency patterns, create all data models, and build empty UI skeleton with system theme support

**Depends on**: Nothing (first phase)

**Requirements**: UI-01, UI-02, UI-03

**Success Criteria** (what must be TRUE):
  1. App launches with dark/light mode following macOS system theme automatically
  2. NavigationSplitView displays empty sidebar, content area, and detail pane with proper macOS layout
  3. All Swift models (Project, Roadmap, Phase, State, Requirement, Plan) exist with Codable conformance
  4. Xcode project compiles with Swift 6 strict concurrency enabled without warnings
  5. Memory profiling with Instruments shows no leaks when navigating between empty views

**Plans**: 3 plans

Plans:

- [x] 01-02-PLAN.md — Build NavigationSplitView skeleton with empty states
- [x] 01-03-PLAN.md — Verify memory safety and theme support

### Phase 2: File Discovery & Parsing
**Goal**: Discover GSD projects in ~/Developer, parse all .planning/ files into Swift models, and persist access with security-scoped bookmarks

**Depends on**: Phase 1

**Requirements**: NAV-01, NAV-02, NAV-05, PARSE-01, PARSE-02, PARSE-03, PARSE-04, PARSE-05

**Success Criteria** (what must be TRUE):
  1. App auto-discovers all .planning/ directories in ~/Developer and displays project names in sidebar
  2. User can manually add project folders outside ~/Developer via file picker
  3. Clicking a project shows parsed roadmap data (phases, goals, requirements) in content area
  4. App correctly parses ROADMAP.md, STATE.md, REQUIREMENTS.md, PLAN.md, and config.json from real GSD projects
  5. Security-scoped bookmarks persist file access across app restarts without re-prompting user

### Phase 3: File System Monitoring
**Goal**: Automatically update project data when .planning/ files change on disk via FSEvents monitoring

**Depends on**: Phase 2

**Requirements**: MON-01, MON-02, MON-03

**Success Criteria** (what must be TRUE):
  1. App detects when ROADMAP.md or STATE.md changes in terminal and updates UI within 1 second
  2. App handles rapid file changes (Git commits, multi-file saves) correctly with event coalescing
  3. App excludes .git/ directories from monitoring to avoid event storms
  4. User sees live phase progress updates when GSD commands run in terminal
  5. FileWatcherService cleanup (stop/invalidate/release) prevents memory leaks verified by Instruments

**Plans**: 3 plans

Plans:

- [x] 03-02-PLAN.md — ProjectService integration with debounced reload

### Phase 5: Notifications & Editor Integration
**Goal**: Send macOS notifications on status changes and open files in user's preferred editor

**Depends on**: Phase 4

**Requirements**: NOTF-01, NOTF-02, NOTF-03, EDIT-01, EDIT-02, EDIT-03

**Success Criteria** (what must be TRUE):

  2. Notifications respect macOS Focus modes and Do Not Disturb
  3. User can click a file path to open it in Cursor, VS Code, or Zed
  4. App auto-detects installed editors in /Applications
  5. User can select preferred editor in Settings
