  <how-to-verify>
**Test 1: Basic live update (ROADMAP.md)**
1. Launch GSD Monitor (Cmd+R in Xcode)
2. Select a project in the sidebar that shows roadmap phases
3. Open a terminal and edit that project's `.planning/ROADMAP.md` (e.g., change a phase goal text)
4. Save the file
5. **Expected:** Within ~2 seconds, the detail view updates to show the changed text. No manual refresh needed.

**Test 2: State change detection (STATE.md)**
1. With the same project selected, open `.planning/STATE.md` in terminal
2. Change the "Phase:" line (e.g., increment the plan number)
3. Save the file
4. **Expected:** Within ~2 seconds, the detail view reflects the state change.

**Test 3: Event coalescing (rapid changes)**
1. In terminal, run a rapid sequence of file touches:
   ```bash
   for f in ROADMAP.md STATE.md config.json; do
     touch /path/to/project/.planning/$f
     sleep 0.1
   done
   ```
2. **Expected:** The UI updates ONCE (not three times). You should NOT see flickering or multiple rapid refreshes.

**Test 4: .git/ exclusion**
1. In a project directory that is a Git repo, make a Git commit (e.g., `echo test >> /tmp/test && git add -A && git commit -m "test"`)
2. **Expected:** The app does NOT show unnecessary reloads from .git/ internal file changes. Only .planning/ changes trigger updates.

**Test 5: Memory check**
1. Open Instruments (Xcode -> Open Developer Tool -> Instruments)
2. Select the "Leaks" template
3. Attach to the running GSD Monitor process
4. Trigger 5-10 file changes in .planning/ directories
5. **Expected:** No new leaks appear in the Leaks timeline. FSEventStream objects are properly cleaned up.

**Test 6: Add project + monitoring**
1. Use the + button to manually add a new project
2. Edit that project's .planning/ROADMAP.md in terminal
3. **Expected:** The new project's changes are detected and UI updates.
