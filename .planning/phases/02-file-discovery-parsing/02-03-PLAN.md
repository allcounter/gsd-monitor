  <action>
    Follow RED-GREEN-REFACTOR TDD cycle for each parser below.

    **RequirementsParser** — input: REQUIREMENTS.md string content -> output: [Requirement]

    Behavior:
    The REQUIREMENTS.md format (reference: this project's file):
    - H3 headers define categories: "### Navigation", "### Roadmap", etc.
    - Under each category, bullet items follow pattern: `- [ ] **REQ-ID**: Description` or `- [x] **REQ-ID**: Description`
    - [x] = validated/done, [ ] = active
    - Traceability table at bottom maps Requirement -> Phase -> Status

    Parser must extract:
    - `id`: The REQ-ID string (e.g., "NAV-01", "PARSE-03")
    - `category`: Current H3 heading (e.g., "Navigation", "Parsing")
    - `description`: Text after the REQ-ID colon
    - `status`: .validated if [x], .active if [ ]
    - `mappedToPhases`: From traceability table (e.g., "Phase 2" -> [2])

    Test cases:
    - Parse real REQUIREMENTS.md from this project -> 29 requirements extracted
    - Parse requirements with mixed [x]/[ ] statuses
    - Parse category grouping correctly (NAV-01 category = "Navigation")
    - Parse traceability table for phase mapping
    - Parse empty REQUIREMENTS.md -> empty array
    - Parse file with no traceability table -> mappedToPhases = []

    Implementation:
    Create `struct RequirementsParser` with `func parse(_ content: String) -> [Requirement]`. Use MarkupWalker:
    - Track current H3 heading as category
    - On list items: match pattern `**{ID}**:` or `**{ID}**: {description}` using regex
    - Check parent checkbox state: look for `[x]` or `[ ]` in raw text of ListItem
    - For traceability: after encountering "## Traceability" heading, parse table rows matching `| REQ-ID | Phase N | Status |`
    - Build Requirement structs, merge traceability mappedToPhases with previously extracted requirements by ID

    **PlanParser** — input: PLAN.md string content -> output: Plan struct

    Behavior:
    The PLAN.md format (reference: this project's 01-01-PLAN.md):
    - YAML frontmatter with phase, plan, type, wave, depends_on, files_modified, autonomous, must_haves
    - <objective> section with plan description
    - <tasks> section containing <task> elements with <name>, <action>, <verify>, <done>
    - Task type attribute: type="auto" or type="checkpoint:human-verify"

    Parser must extract:
    - `phaseNumber`: from frontmatter "phase: XX-name" -> extract XX as Int
    - `planNumber`: from frontmatter "plan: NN"
    - `objective`: text content of <objective> section
    - `tasks`: array of Task structs from <task> elements
      - `name`: from <name> element, strip "Task N: " prefix
      - `type`: .auto for type="auto", .checkpoint for type="checkpoint:*"
      - `status`: default .pending (PLAN.md doesn't track execution status)

    NOTE: PLAN.md uses XML-like tags inside markdown. Parse these with string matching (regex), not the markdown AST, since they're not standard markdown elements. The frontmatter is YAML between --- delimiters.

    Test cases:
    - Parse real 01-01-PLAN.md from this project -> Plan with 2 tasks
    - Parse plan with checkpoint task -> task.type = .checkpoint
    - Parse frontmatter phase "02-file-discovery-parsing" -> phaseNumber = 2
    - Parse plan with no tasks -> empty tasks array
    - Parse objective text correctly
    - Handle malformed frontmatter gracefully

    Implementation:
    Create `struct PlanParser` with `func parse(_ content: String) -> Plan?`. Use string-based parsing (not MarkupWalker) since PLAN.md mixes YAML frontmatter with XML-like task tags:
    - Extract frontmatter between first `---` pair. Parse key-value pairs.
    - Extract phase number: regex on `phase: (\d+)-` or `phase: (\d+)`
    - Extract plan number: `plan: (\d+)`
    - Extract objective: content between `<objective>` and `</objective>` tags
    - Extract tasks: find all `<task type="...">` blocks. For each:
      - Extract type attribute value
      - Extract <name> content
      - Extract <action>, <verify>, <done> content (store in task name for now; Task model is simple)
    - Return Plan struct. Return nil if content is unparseable.

    **Test setup:**
    Use XCTest with embedded string literals for test data. Reference the actual files for realistic formats but don't read from disk in tests.
