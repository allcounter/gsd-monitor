  <action>
    Follow RED-GREEN-REFACTOR TDD cycle for each parser below.

    **RoadmapParser** — input: ROADMAP.md string content -> output: Roadmap struct

    Behavior:
    - Extracts project name from H1 "# Roadmap: {name}"
    - Extracts phases from H3 "### Phase N: Name" headers (note: ROADMAP.md uses H3 for phase details, H2 for section headers like "Phase Details")
    - For each phase: parses Goal (bold text after **Goal**:), Depends on, Requirements, Success Criteria
    - Detects completion status from checkbox: `- [x]` = done, `- [ ]` = not started/in progress
    - Uses Phase.status based on: `[x]` = .done, plus plans completion for inProgress detection
    - Returns Roadmap with projectName and phases array

    Test cases:
    - Parse real ROADMAP.md content from this project (use .planning/ROADMAP.md as reference)
    - Parse minimal ROADMAP with only 1 phase
    - Parse empty ROADMAP (no phases) -> returns Roadmap with empty phases
    - Parse ROADMAP with missing Goal section -> phase.goal = ""
    - Parse ROADMAP with [x] completed phases -> status = .done

    Implementation:
    Create `struct RoadmapParser` with `func parse(_ content: String) -> Roadmap`. Use swift-markdown's `Document(parsing:)` and a custom `MarkupWalker` subclass. Walk the AST:
    - On H1: extract project name after "Roadmap: "
    - On H3 "Phase N: Name": create new phase builder
    - On bold text "**Goal**:": next content is goal
    - On ordered/unordered lists after specific headers: extract items
    - Use regex `/Phase (\d+): (.+)/` for phase number/name extraction (defensive parsing per research pitfall 5)
    - Build Phase structs with extracted data. Default status to .notStarted, set .done if "[x]" found in phase list item.

    **StateParser** — input: STATE.md string content -> output: State struct

    Behavior:
    - Extracts "Phase: N of M" for currentPhase
    - Extracts "Plan: N of M" for currentPlan
    - Extracts "Status:" line for status string
    - Extracts "Last activity:" for lastActivity
    - Extracts decisions from "### Decisions" bullet list
    - Extracts blockers from "### Blockers/Concerns" bullet list

    Test cases:
    - Parse real STATE.md from this project
    - Parse STATE.md with "None yet." blockers -> empty blockers array
    - Parse STATE.md with multiple decisions -> all captured
    - Parse minimal STATE.md with only status line

    Implementation:
    Create `struct StateParser` with `func parse(_ content: String) -> State`. Use MarkupWalker to extract:
    - Paragraph text matching "Phase: X of Y" -> currentPhase = X
    - Paragraph text matching "Plan: X of Y" -> currentPlan = X
    - "Status:" line -> status string
    - List items under "### Decisions" heading -> decisions array
    - List items under "### Blockers/Concerns" heading -> blockers array
    - Handle "None yet." by returning empty array

    **ConfigParser** — input: config.json Data -> output: PlanningConfig struct

    Behavior:
    - Uses JSONDecoder with .convertFromSnakeCase strategy
    - PlanningConfig already exists in Project.swift but only has workflowVersion and autoCommit
    - Extend or update PlanningConfig to include: mode (String?), depth (String?), parallelization (Bool?)
    - Falls back gracefully on missing/extra keys (all properties optional)

    NOTE: The existing PlanningConfig in Project.swift uses manual CodingKeys for snake_case. For the new fields, either add them with CodingKeys or switch to .convertFromSnakeCase strategy. Since existing fields use manual CodingKeys, add new fields with CodingKeys to stay consistent.

    Test cases:
    - Parse real config.json from this project
    - Parse config.json with missing fields -> nil optionals
    - Parse empty JSON object -> all nil
    - Parse config.json with extra unknown fields -> ignores them

    Implementation:
    Create `struct ConfigParser` with `func parse(_ data: Data) throws -> PlanningConfig`. Use JSONDecoder. Update PlanningConfig model to include mode, depth, parallelization fields with CodingKeys.

    **Test setup:**
    Create a GSDMonitorTests target if it doesn't exist. Each test file uses XCTest. Embed test markdown/JSON as string literals (not file reads) for reliability. Use the real project files as reference for realistic test data.
